## 1. Purpose

`mobman_moveit_config` is the **MoveIt 2 configuration and launch package** for the CHARS mobile manipulator (MOBMAN). It was generated by the MoveIt Setup Assistant and then extended with namespace support and a Python motion test utility.

The package provides everything needed to bring up a fully functional MoveIt planning stack for the `mobile_manipulator` robot model:

- Robot semantic description (SRDF) — planning groups, named states, virtual joints, passive joints, and ACM (Allowed Collision Matrix).
- Kinematics solver configuration.
- Joint limits and scaling factors.
- ros2_control hardware interface and controller definitions.
- MoveIt controller manager mapping.
- Pilz cartesian planner limits.
- Launch files for every sub-component of the MoveIt stack.
- A Python motion test script for integration validation.

### Role in CHARS

In the CHARS framework, this package serves **Layer 1 (Execution & Perception)**. The CHARS task action server (`mobman_task_action_server`) communicates with the `move_group` node started by this package to plan and execute arm trajectories for Pick and Place operations. Every time a robot needs to move its arm to a pick standoff, perform an IK-probing sweep, or place a component, the MoveIt stack defined here is responsible for that motion.

## 4. Robot Model

### Overview

The `mobile_manipulator` is a composite robot:

- **Mobile base**: Jackal (differential-drive, 4-wheel skid-steer)
- **Arm**: Custom 6-DOF manipulator mounted on a plate on the Jackal chassis

### Planning Group: `mobman_arm`

The single defined planning group covers the entire kinematic chain from world to the arm tip.

| Joint | Type | Description |
|---|---|---|
| `world_joint` | Virtual / fixed | Anchors `chassis_link` to the world frame |
| `arm_mount_joint` | Fixed (passive) | Mount plate to chassis |
| `arm_mount_base_joint` | Fixed (passive) | Mount plate to arm base |
| `arm_base_link1_joint` | Revolute | Shoulder / base rotation (J1) |
| `arm_link1_link2p1_joint` | Revolute | Upper arm (J2) |
| `arm_p1_p2_joint` | Revolute | Elbow segment 1 (J3 link part) |
| `arm_p2_p3_joint` | Revolute | Elbow segment 2 (J3 link part) |
| `arm_link2p3_link3_joint` | Revolute | Forearm fold (J3) |
| `arm_link3_link4_joint` | Revolute | Wrist 1 (J4) |
| `arm_link4_link5_joint` | Revolute | Wrist 2 (J5) |
| `arm_link5_link6_joint` | Revolute | Wrist 3 / end-effector rotation (J6) |

**Actuated DOF:** 6 active revolute joints (`arm_base_link1_joint` through `arm_link5_link6_joint`)

**Passive joints (wheels):** `front_left_wheel`, `front_right_wheel`, `rear_left_wheel`, `rear_right_wheel` — excluded from planning.

**End-effector link:** `arm_link6_1`

### Named Group States

| State Name | Group | Joint Values (rad) |
|---|---|---|
| `home` | `mobman_arm` | `[0, 0, 0, 0, 0, 0]` — all joints at zero |
| `vertical` | `mobman_arm` | `[0, 0, 1.57, 1.57, 0, 0]` — arm pointing upward |

### Virtual Joint

```
world_joint: fixed  →  parent: world  →  child: chassis_link
```

## 5. Configuration Files

### `kinematics.yaml`

Defines the IK solver for the `mobman_arm` planning group.

| Parameter | Value | Description |
|---|---|---|
| `kinematics_solver` | `kdl_kinematics_plugin/KDLKinematicsPlugin` | KDL numerical IK solver |
| `kinematics_solver_search_resolution` | `0.005` rad | Angular step size during IK search |
| `kinematics_solver_timeout` | `0.005` s | Maximum time per IK solve attempt |

> **Note:** The 5ms timeout is aggressive. For complex pick poses that require reachability probing, the CHARS task action server calls `compute_ik` with a longer timeout override. Consider increasing to `0.05`–`0.1` s if IK failures occur during fast replanning.

---

### `joint_limits.yaml`

Overrides URDF-specified dynamics and sets global planning speed scalars.

| Parameter | Value | Description |
|---|---|---|
| `default_velocity_scaling_factor` | `0.1` (10%) | Global velocity ceiling applied to all motion requests |
| `default_acceleration_scaling_factor` | `0.1` (10%) | Global acceleration ceiling |

All 6 actuated joints have `has_velocity_limits: false` and `has_acceleration_limits: false` — meaning MoveIt relies entirely on the scaling factors above and the URDF limits (or ros2_control limits), not per-joint overrides from this file.

> **Practical note for CHARS:** The action server can override these scalars per-request via `max_velocity_scaling_factor` and `max_acceleration_scaling_factor` on the `MoveGroup.Goal`. The test script uses `0.5` (50%) by default.

---

### `moveit_controllers.yaml`

Maps MoveIt's trajectory execution manager to the ros2_control `JointTrajectoryController`.

```yaml
moveit_controller_manager: moveit_simple_controller_manager/MoveItSimpleControllerManager

moveit_simple_controller_manager:
  controller_names:
    - mobman_arm_controller

  mobman_arm_controller:
    type: FollowJointTrajectory
    action_ns: follow_joint_trajectory
    default: true
    joints:
      - arm_base_link1_joint
      - arm_link1_link2p1_joint
      - arm_link2p3_link3_joint
      - arm_link3_link4_joint
      - arm_link4_link5_joint
      - arm_link5_link6_joint
```

MoveIt sends planned trajectories to the `follow_joint_trajectory` action server exposed by `mobman_arm_controller`.

## 7. Topics and Actions

All topics are namespaced under `/<robot_namespace>/` (default: `/mobman/`).

### Topics Published by `move_group`

| Topic | Type | Description |
|---|---|---|
| `/<ns>/move_group/monitored_planning_scene` | `moveit_msgs/PlanningScene` | Live planning scene state, including known obstacles |
| `/<ns>/move_group/status` | `actionlib_msgs/GoalStatusArray` | Status of active motion goals |
| `/tf`, `/tf_static` | `tf2_msgs/TFMessage` | Transform tree (via `robot_state_publisher`) |
| `/<ns>/joint_states` | `sensor_msgs/JointState` | Broadcast by `joint_state_broadcaster` (ros2_control) |

### Topics Subscribed by `move_group`

| Topic | Type | Description |
|---|---|---|
| `/<ns>/joint_states` | `sensor_msgs/JointState` | Used to update robot current state for planning |
| `/<ns>/robot_description` | `std_msgs/String` | URDF for kinematic model |

### Action Servers Provided by `move_group`

| Action | Type | Description |
|---|---|---|
| `/<ns>/move_action` | `moveit_msgs/action/MoveGroup` | **Primary planning + execution interface.** Accepts goal constraints (joint or pose), plans a trajectory, and optionally executes it. Used by both the test script and the CHARS task action server. |
| `/<ns>/execute_trajectory` | `moveit_msgs/action/ExecuteTrajectory` | Execute a pre-planned trajectory directly |
| `/<ns>/compute_ik` | (service) `moveit_msgs/srv/GetPositionIK` | Inverse kinematics query |
| `/<ns>/compute_fk` | (service) `moveit_msgs/srv/GetPositionFK` | Forward kinematics query |
| `/<ns>/get_planning_scene` | (service) `moveit_msgs/srv/GetPlanningScene` | Query the current planning scene |
| `/<ns>/apply_planning_scene` | (service) `moveit_msgs/srv/ApplyPlanningScene` | Modify the planning scene (add/remove objects) |
| `/<ns>/clear_octomap` | (service) `std_srvs/srv/Empty` | Clear the OctoMap occupancy grid |

## 8. Launch Files

### 8.1 `mobman_moveit.launch.py` — Primary (CHARS Integration)

The main launch file used in the CHARS swarm setup. Starts `move_group` inside a robot namespace to support multiple simultaneous robot instances.

**Launch arguments:**

| Argument | Default | Description |
|---|---|---|
| `robot_namespace` | `mobman` | Namespace for the `move_group` node and all ROS topics. Must match the namespace used by `robot_state_publisher` and the ros2_control instance for that robot. |
| `use_sim_time` | `true` | Use Gazebo simulation clock (`/clock`) |
| `use_rviz` | `true` | Launch RViz2 with MoveIt plugin (**currently commented out** in the return statement — see Known Issues) |
| `allow_trajectory_execution` | `true` | Whether `move_group` is allowed to send trajectories to the robot's controller |
| `publish_monitored_planning_scene` | `true` | Publish geometry, state, and transform updates for the planning scene |

**Usage:**

```bash
# Default namespace (mobman)
ros2 launch mobman_moveit_config mobman_moveit.launch.py

# Custom namespace (for multi-robot CHARS setup)
ros2 launch mobman_moveit_config mobman_moveit.launch.py robot_namespace:=agent1

# Without RViz
ros2 launch mobman_moveit_config mobman_moveit.launch.py use_rviz:=false
```

**⚠️ Known Issue:** The `rviz_node` is commented out in the `LaunchDescription` return statement. Also, the RViz config path points to `ur5_moveit_config` instead of `mobman_moveit_config`. If RViz is needed, fix line 128 of this launch file:
```python
# Change:
FindPackageShare("ur5_moveit_config"),
# To:
FindPackageShare("mobman_moveit_config"),
# And uncomment rviz_node in the return LaunchDescription list.
```

## 9. Test Script: `test_mobman_moveit.py`

### Node: `moveit_motion_tester`

A standalone Python ROS 2 node (`scripts/test_mobman_moveit.py`) for validating the full MoveIt stack. Installed as a `console_scripts` entry point.

#### Parameters

| Parameter | Type | Default | Description |
|---|---|---|---|
| `robot_namespace` | `string` | `mobman` | Robot namespace prefix; used to build topic and action names |
| `planning_group` | `string` | `mobman_arm` | MoveIt planning group name |
| `execute` | `bool` | `true` | If `false`, plan only — do not send trajectory to controller |
| `target_pose` | `string` | `ready` | Named pose to move to (see predefined poses below) |
| `use_cartesian` | `bool` | `false` | If `true`, use cartesian XYZ goal instead of named pose |
| `target_x` | `double` | `0.0` | Cartesian target X (m) in world frame |
| `target_y` | `double` | `0.0` | Cartesian target Y (m) in world frame |
| `target_z` | `double` | `0.0` | Cartesian target Z (m) in world frame |
| `end_effector_link` | `string` | `arm_link6_1` | End-effector link name for cartesian goals |
| `planning_time` | `double` | `5.0` | Allowed planning time per attempt (s) |

#### Predefined Poses

| Pose Name | Joint Values (rad) | Description |
|---|---|---|
| `home` | `[0, 0, 0, 0, 0, 0]` | All joints at zero (flat/stowed) |
| `ready` | `[1.57, 0, 0, 0, 0, 0]` | Base rotated 90°, arm forward |
| `up` | `[0, -1.57, 0, 0, 0, 0]` | Arm pointing upward |
| `left` | `[1.57, -0.5, 1.0, -0.5, 0, 0]` | Arm reaching left |
| `right` | `[-1.57, -0.5, 1.0, -0.5, 0, 0]` | Arm reaching right |

#### Topics / Actions Used

| Interface | Name | Description |
|---|---|---|
| Subscriber | `/<ns>/joint_states` | `sensor_msgs/JointState` — reads current arm state |
| Action Client | `/<ns>/move_action` | `moveit_msgs/action/MoveGroup` — sends planning + execution goals |

#### Motion Goal Configuration

**Joint goals:**
- `num_planning_attempts`: 10
- `max_velocity_scaling_factor`: 0.5
- `max_acceleration_scaling_factor`: 0.5
- `replan_attempts`: 3
- Joint tolerance: ±0.01 rad per joint
- Workspace bounds: ±2.0 m in all axes

**Cartesian goals:**
- `max_velocity_scaling_factor`: 0.3
- `max_acceleration_scaling_factor`: 0.3
- Position tolerance: 1 cm radius sphere
- Orientation tolerance: ±0.1 rad per axis
- Default orientation: `qy=0.707, qw=0.707` (pointing forward-down)

#### Usage Examples

```bash
# Move to default 'ready' pose
ros2 run mobman_moveit_config test_mobman_moveit

# Move to a specific named pose
ros2 run mobman_moveit_config test_mobman_moveit --ros-args -p target_pose:=up

# Cartesian target (with execution)
ros2 run mobman_moveit_config test_mobman_moveit --ros-args \
  -p use_cartesian:=true -p target_x:=0.4 -p target_y:=0.0 -p target_z:=0.5

# Plan only — inspect the planned path without executing
ros2 run mobman_moveit_config test_mobman_moveit --ros-args -p execute:=false

# Custom robot namespace (e.g., for agent1 in multi-robot CHARS setup)
ros2 run mobman_moveit_config test_mobman_moveit --ros-args -p robot_namespace:=agent1
```

## 12. Troubleshooting

| Symptom | Likely Cause | Fix |
|---|---|---|
| `move_group` not available — action not found | Namespace mismatch or move_group not started | Check `robot_namespace` matches how `robot_state_publisher` was launched; verify with `ros2 action list` |
| No robot state / TF errors in move_group | `robot_state_publisher` or `joint_state_broadcaster` not running | Confirm Gazebo or fake hardware started; check `ros2 topic echo /<ns>/joint_states` |
| `No IK solution found` | Target pose out of workspace or tight IK timeout | Increase `kinematics_solver_timeout` in `kinematics.yaml`; verify target is within arm reach |
| Planning always fails | Start state in collision or goal in collision | Check the planning scene; ensure the robot starts at a valid state (`home` pose) |
| `arm_link6_1` frame not found during cartesian test | TF frame prefix not matching | Confirm `end_effector_link` parameter matches the TF frame published by RSP; in namespaced setup it may be `agent1/arm_link6_1` |
| RViz config not loading | Hardcoded `ur5_moveit_config` path in `mobman_moveit.launch.py` | Update line 128 to `FindPackageShare("mobman_moveit_config")`; also uncomment `rviz_node` in the return list |
| Controllers not spawning | `controller_manager` not running or URDF hw interface missing | Ensure `ros2_control` tag is in the URDF (via `mobile_manipulator.ros2_control.xacro`); check `ros2 control list_controllers` |
| Joint velocity/acceleration limits ignored | `has_velocity_limits: false` in `joint_limits.yaml` | Either enable per-joint limits in `joint_limits.yaml` or use `max_velocity_scaling_factor` overrides in the motion goal |

## 13. Integration Notes for CHARS

- **Namespace convention:** In CHARS, each Jackal+UR5 robot instance runs under its own namespace (e.g., `agent1`, `agent2`). Launch `mobman_moveit.launch.py` once per agent, passing `robot_namespace:=<agent_name>`. The `move_group` action, planning scene service, and all IK services will be scoped to that namespace.
- **Planning frame:** The SRDF defines `world_joint` as a fixed virtual joint from `world` to `chassis_link`. When the robot moves via Nav2, the `world` → `chassis_link` transform changes, so the planning scene must be updated or replanned to reflect the new base pose. The CHARS action server handles this by re-querying TF before submitting arm motion goals.
- **Collision geometry:** The SRDF ACM (Allowed Collision Matrix) disables self-collision checking for all structurally adjacent and mechanically constrained link pairs. It does **not** include environment collision objects — these must be added to the planning scene at runtime via `/apply_planning_scene` if needed.
- **Hardware plugin:** The `mobile_manipulator.ros2_control.xacro` uses `mock_components/GenericSystem` by default. For Gazebo Fortress, the `swarm_description` URDF replaces this plugin with the Gazebo ros2_control hardware interface automatically.
- **IK solver note:** KDL is a numerical solver and can fail for degenerate configurations or near joint limits. If IK success rates are low during pick actions, consider switching to `trac_ik_kinematics_plugin/TRAC_IKKinematicsPlugin` in `kinematics.yaml` for better coverage.
